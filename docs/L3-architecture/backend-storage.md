## Module Detail: `durable-run-storage`

**Purpose:** Manages the persistent lifecycle of subagent runs under `.pi/subagents/runs/` — creates run directories, writes and reads JSONL event/tool/session logs, searches across runs, formats run listings, and applies retention cleanup policies.

### Member L1 Files

| L1 Doc | Summary |
|--------|---------|
| `storage-event-log.md` | Core I/O layer for all durable run artifacts. Creates run directories with deterministic naming (`{profile}-{task}-{timestamp}-{hexSuffix}`), manages `session.json` state, appends typed entries to `events.jsonl` / `session.jsonl` / `tools.jsonl`, reads back events, searches runs by filters (`RunSearchQuery`), formats human-readable run listings, and cleans up old runs per `CleanupPolicy`. |

### Key Exports

- `RunDirectory` — Path bundle: `{runDir, sessionPath, eventsPath, toolsPath, handoffPath}`
- `createRunDir(...)` — Creates a run directory with deterministic naming
- `readEvents(run: RunDirectory)` — Reads events from JSONL
- `appendEvent(run, entry)` / `appendToolLog(run, entry)` / `appendSessionMessage(run, entry)` — Write operations
- `searchRuns(query: RunSearchQuery)` — Filter/search across runs
- `listRuns()` / `formatRunList(runs)` — Listing and display
- `cleanupRuns(policy: CleanupPolicy)` — Retention cleanup

### Data Model

```
.pi/subagents/runs/
└── {profile}-{task}-{timestamp}-{hexSuffix}/
    ├── session.json       ← run metadata (profile, task, timestamps, status)
    ├── events.jsonl       ← typed event log (run_start, tool_call, hook_exec, policy_block, ...)
    ├── tools.jsonl        ← tool execution results with inputs/outputs
    └── handoff.md         ← structured handoff document (generated by Run Artifact Generation)
```

### Why This Classification

`durable-run-storage` is the **canonical storage module**. Every function it exports is about persistence: create directories, write JSONL entries, read data back, search indexed runs, format listings, clean up old data. It has no computation beyond formatting run names and sorting listings. It has no display — `formatRunList()` produces data for display, not rendered terminal output. It is the single source of truth for all run data persistence.

## Module Detail: `registry-storage`

**Purpose:** Layer 1 of the Prompt Registry — persistent storage and in-memory indexing. Owns the `registry.jsonl` (project-level, full-rewrite) and `registry-calls.jsonl` (per-run, append-only) files, maintains four O(1) lookup indexes, and tracks per-entry sliding-window frequency counters.

### Member L1 Files

| L1 Doc | Summary |
|--------|---------|
| `registry-storage.md` | `RegistryStorage` class: JSONL-backed CRUD, tag/group/name indexing, call-history recording, sliding-window frequency tracking, and serialization/deserialization of frequency state for session resume. |

### Key Exports

- `RegistryStorage` class — Main storage interface
  - **CRUD:** `register(entry)`, `update(id, entry)`, `unregister(id)`, `get(id)`, `getByName(name)`
  - **Queries:** `list()`, `findByTags(tags)`, `findByGroup(group)`
  - **Call tracking:** `recordCall(entryId)`, `getFrequency(entryId)`, `getTotalCalls(entryId)`
  - **Lifecycle:** `load()` (rebuilds indexes from JSONL), `save()` (full rewrite)
  - **Serialization:** `exportFreqState()`, `loadFreqState(data)` for cross-session continuity

### Internal Architecture

```
registry.jsonl ──load()──→ [IdIndex, NameIndex, TagIndex, GroupIndex] ──→ CRUD + queries
registry-calls.jsonl ──recordCall()──→ [SlidingWindowCounter per entry] ──→ getFrequency()
```

- **Indexes:** Four `Map`-based O(1) lookup structures rebuilt on `load()`, updated transactionally on mutations
- **Frequency:** Private `SlidingWindowCounter` class driving public frequency/count queries
- **Write strategy:** `registry.jsonl` → full rewrite on `save()`; `registry-calls.jsonl` → append-only on `recordCall()`

### Why This Classification

`registry-storage` is a **pure storage module**. Despite having indexes and frequency counters (which could be seen as computation), these are infrastructure for efficient storage access — not business logic. The module's purpose is to persist registry entries, index them for fast lookup, and track usage frequency. All operations are CRUD + query; the "computation" (frequency counting) is an implementation detail of the storage mechanism, not the module's purpose.

## Layer Position in Architecture

```
┌──────────────────────────────────────────────────┐
│                  BACKEND                          │
│  ┌──────┐ ┌───────┐ ┌──────────────────────────┐ │
│  │ 输入  │ │ 输出   │ │  存储 (Storage)          │ │
│  └──────┘ └───────┘ │  durable-run-storage     │ │
│                     │  registry-storage        │ │
│                     └──────────┬───────────────┘ │
│  ┌────────────────────────────┐│                 │
│  │ 计算                        │▼                 │
│  │ (consumes storage queries)  │                  │
│  └────────────────────────────┘                  │
└──────────────────────────────────────────────────┘
```

The 存储 layer is the **persistence foundation** — it provides data durability, indexing, and lifecycle management for all other layers. Computation modules query storage; output modules read from storage; input modules don't directly interact with storage (they feed into computation which then writes through storage APIs). Both modules are self-contained and have no external dependencies within the extension.
