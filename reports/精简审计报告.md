## 三、被反驳的发现（Refuted — 5 项）

### DC-4: context-mounts.ts 的冗余动态导入
- **原判断**: 第 1-5 行存在 session-store.ts 的静态导入，line 47 的动态导入冗余。
- **实际验证**: **静态导入不存在**。context-mounts.ts 中只有 line 47 一个 `await import("../../core/session-store.ts")` 动态导入。原前提错误，该动态导入是合理的（避免循环依赖）。
- **结论**: 被反驳。代码正确，无需修改。

### OE-7: 7 个 action 文件的 try-catch 模式重复
- **原判断**: 每个 action 文件有 1 个相同模式的 try-catch-ok/error 块，可提取共享 helper。
- **实际验证**: `for f in tool/actions/*.ts; do grep -c 'try {' $f; done` 显示只有 4 个文件有恰好 1 个 `try`：archive-session(1)、list-models(1)、read-events(1)、update-session(1)；其余 3 个文件各有多个 try（context-mounts 3 个、create-session 3 个、send-task 2 个），包含独特/额外的错误处理逻辑，不适用于共享 helper。
- **结论**: 被反驳。handler 间的错误处理有差异化逻辑。

### OE-8: TypeBox schema 被 `as any` 绕过
- **原判断**: `as any` 转型导致 20 行 TypeBox 运行时验证代码失效。
- **实际验证**: `as any` 是 **TypeScript 编译期类型断言**，在运行时完全无影响。完整的 TypeBox schema 对象仍然被构建并传递给 PI SDK 的 `registerTool`。如果 PI SDK 在运行时进行 schema 验证，它完全能正常工作。`as any` 只是抑制了 TypeScript 的类型检查（常见于第三方 SDK 类型定义与 TypeBox 不兼容的场景）。
- **结论**: 被反驳。运行时验证不受影响。但 schema 的编译时类型安全确实丢失，建议移除 `as any` 并找到正确的类型表达。

### CO-5: handleListModels 参数签名反转
- **原判断**: `handleListModels` 使用 `(ctx, params?)` 而非其他 handler 的 `(params, ctx)`，router 交换参数调用。
- **实际验证**: router（line 72）调用 `handleListModels(ctx, params as any)` —— 与函数签名的 `(ctx, params?)` **完全一致**，不存在参数交换。不一致仅存在于函数签名约定层面，router 调用是正确的。
- **结论**: 被反驳。没有运行时 bug。但签名确实与其他 handler 不一致，建议统一为 `(params, ctx)` 以降低未来重构风险。

### OE-10: ?? [] 是 TypeScript 类型正确性的必要条件
- **原判断**: line 138 的 `.filter(mount => mount.sources?.length)` 保证了 `sources` 不为空，line 139 的 `?? []` 冗余。
- **实际验证**: `Array.prototype.filter` 的回调未使用类型谓词，TypeScript **无法通过 filter 推断嵌套属性的类型收窄**。Filter 后 `mount.sources` 仍为 `ContextSource[] | undefined`；没有 `?? []` 则 `.flatMap()` 会因为 `undefined` 不可迭代而报类型错误。代码当前写法是正确的，移除 `?? []` 会导致编译失败。
- **结论**: 被反驳。原判断是虚假阳性。如需真正消除 `?? []`，需添加类型谓词 filter（比现状更多代码）。
